# 苏州大学导师信息检索系统 8-10 分钟演讲稿

本文档对应 21 页 PPT，由一个人完成汇报，目标时长约 8-10 分钟。讲稿中的实验结果采用项目文档 `docs/ir_model_handoff.md`，具体可以更换成最新实验结果的。

> 数值口径：本文中的数据规模和指标已于 2026-06-13 从本地 JSONL 与 `scripts/evaluate_search.py` 重新统计/复跑验证。相关性指标可直接汇报；耗时会受机器、缓存、BGE 模型加载状态影响，建议只作为量级参考。

建议汇报时反复扣住一句主线：本项目不是简单关键词搜索，而是面向真实导师主页数据，完成了结构化数据构建、自适应混合检索、条件式语义增强和可量化消融评测的一套垂直 IR 系统。

## 汇报总策略

汇报时不要从“我们写了哪些文件”开始讲，而要按问题推进：

1. 真实教师主页数据不规整，所以先做结构化解析和清洗。
2. 有了统一教师文档后，先建立 BM25 baseline，作为可量化对照。
3. baseline 对简单查询可用，但对学院约束、口语化 query、论文成果 query 不够稳定。
4. 因此设计 optimized：查询意图识别、动态字段权重、查询扩展、模糊姓名、学院 hard filter 和 paper-only RRF。
5. 进一步探索 BGE 语义召回，但发现不能无条件替代 BM25，于是做成 conditional_semantic。
6. 最后用 48 条主测试 query、216 条主 qrels，以及 10 条语义泛化 query、65 条语义 qrels 做量化验证。

这条线能同时凸显工作量和思考过程：我们不是只把功能堆起来，而是在每一步发现问题、做工程取舍、再用实验验证。

## 第 1 页：封面

各位老师、同学大家好。我们小组汇报的项目是“面向苏州大学老师信息的垂直检索系统”。

这个项目从苏州大学教师主页出发，完成了网页数据采集、数据清洗、检索模型构建、优化检索、量化评测和前后端展示。我们汇报的重点不是“做了一个搜索框”，而是如何把真实噪声网页数据变成可检索文档，并围绕不同查询意图设计、验证一套自适应 IR 系统。

## 第 2 页：项目目标与核心问题

我们要解决的问题是：如何让用户更方便地检索苏州大学老师信息。

普通搜索引擎虽然可以搜网页，但它不一定适合老师查找场景。用户更关心的是：谁研究某个方向，某个学院有哪些相关老师，某位老师有什么论文成果，以及输入比较口语化的 query 时，系统能否返回真正相关的老师。

所以我们的目标是构建一个面向苏州大学老师信息的垂直检索系统，支持按姓名、学院、研究方向和论文成果等方式查询。

## 第 3 页：系统整体架构

这是系统整体架构。

数据首先来自教师主页 HTML，经过网页解析和数据清洗后，形成统一的教师 JSONL 文档。之后进入索引和检索模型层。

这里我们有两条主要检索路线：一条是 BM25 baseline，作为基础对照模型；另一条是我们的主模型，也就是自适应混合检索 optimized。它在字段级 BM25 基础上，加入查询意图识别、动态字段权重、查询扩展、模糊姓名匹配、学院 hard filter 和论文成果类 RRF 融合。

此外，我们还实现了一个离线评测用的 conditional_semantic 模式。它不是直接用 BGE embedding 替代 BM25，而是在 optimized 稳定排序基础上，通过 semantic gate 判断 query 是否需要语义泛化，只有触发时才引入 BGE 补充召回。

最后，系统通过 FastAPI 后端提供接口，由 Vue 前端展示结果，并通过评测集验证效果.

## 第 4 页：数据来源与规模

我们的数据来自苏州大学教师主页，覆盖 5 个学院。当前主实验数据包含 283 条教师文档。

每条教师文档以 JSONL 形式存储，核心字段包括姓名、学院、职称、研究方向、论文成果、个人简介和全文内容。

为了评估检索效果，我们还构建了正式评测集，包括 48 条 query 和 216 条 qrels 相关性标注。为了验证 BGE 对语义改写类 query 的价值，我们还补充了 10 条语义泛化 query 和 65 条语义 qrels。

这几个数字建议在汇报中明确说出来，因为它们能体现工作量：我们的数据不是几条样例，评测也不是主观看几个结果，而是有 query 和 qrels 的量化比较。

## 第 5 页：数据采集与解析策略

数据采集采用 seed 驱动方式。因为动态网页爬取和自动发现教师主页的难度较大，

主要原因是学校入口页不一定能从静态 HTML 中直接解析出完整教师列表，而且不同学院页面结构差异比较大。相比完全自动发现，维护教师主页 seed 更稳定，也更方便复现。

这里体现的是一个工程取舍：完全自动化发现所有主页当然更理想，但课程项目更需要稳定复现和可扩展。后续如果扩展新学院，只需要继续维护对应学院的 seed 文件和原始 HTML 目录。

系统会根据 seed 抓取教师详情页 HTML，并解析姓名、学院、职称、研究方向、论文和简介等字段。对于姓名和学院这类关键元数据，我们会使用 seed 信息进行校正，避免被网页标题或导航内容干扰。

## 第 6 页：数据处理困难与解决方案

真实网页数据并不规整，这是这个项目的第一个难点。

我们遇到的问题包括：页面模板不统一，入口页动态加载，编码和跳转复杂，导航栏、页脚和访问量等噪声较多，以及部分教师主页字段缺失。

对应的解决方法包括：分别适配新版栏目页和旧版自由格式页，使用 seed 驱动采集，处理 UTF-8、GBK、GB18030 等编码，清理网页噪声，并用字段别名、seed 校正和全文兜底提升字段可用性。

## 第 7 页：统一教师文档模型

清洗后的数据会被组织成统一的教师文档模型。

这里的关键是，我们没有只保留一整段网页正文，而是把姓名、学院、职称、研究方向、论文成果、个人简介和全文内容分开存储。

这样做的好处是，后续检索可以按字段建模。比如姓名查询更重视 name 字段，研究方向查询更重视 research 字段，论文成果查询更重视 papers 字段。

这也是我们后面做字段加权和动态权重调整的基础。

## 第 8 页：Baseline 检索模型

基础检索模型使用 BM25。

BM25 是信息检索中非常经典的排序方法，所以我们把它作为 baseline。中文分词优先使用 jieba，如果环境中没有 jieba，则用字符 n-gram 作为兜底。

baseline 支持姓名精确匹配，也支持学院、研究方向、论文和职称等字段过滤。同时，我们给不同字段设置了基础权重，其中姓名权重最高，研究方向较高，论文、简介、学院和全文也参与综合排序。

baseline 在这里有两个作用：第一，它是一个能正常工作的基础系统；第二，它是后续优化的对照组。所有优化是否有效，都要和 baseline 指标比较，而不是只看主观展示。

## 第 9 页：Baseline 使用示例

baseline 对简单查询已经可用。

比如查询“周国栋”时，可以通过姓名字段返回对应教师；查询“自然语言处理”或“机器学习”时，可以根据研究方向、论文成果和全文内容进行排序。

但 baseline 的不足也比较明显。比如“计科院做知识图谱的老师”这种 query 同时包含学院别名、研究方向和口语化表达，单一 BM25 权重很难稳定处理所有情况。

因此我们进一步设计了 optimized 检索模型。

这里可以作为一个自然转场：baseline 不是失败，而是让我们看清楚了哪些 query 类型需要更细的策略。

## 第 10 页：为什么需要优化检索

用户 query 的类型差异很大。

“周国栋”是姓名查询，“自然语言处理”是研究方向查询，“数学学院 动力系统”是学院加方向查询，“发表 ACL 论文的导师”则更像论文成果查询。

不同 query 应该关注不同字段。如果所有查询都用同一套字段权重，就会出现排序不稳定的问题。

所以我们的核心思路是：先理解 query 的意图，再选择更合适的字段权重和检索策略。

这也是 optimized 的核心设计思想：不是所有查询都用同一个权重表，而是让检索策略随 query 类型变化。

## 第 11 页：Optimized 检索总体流程

optimized 的流程可以概括为：查询理解、字段加权、补充召回和融合排序。

系统首先分析用户 query，识别学院别名、职称词、论文成果词和口语化表达。比如“计科院”会归一到“计算机科学与技术学院”。

然后系统根据 query 类型调整字段权重，并进行查询扩展和模糊姓名匹配。如果识别到学院，就使用学院 hard filter 限定候选教师。

对于论文成果类 query，系统还会使用 RRF 融合 optimized fielded BM25 和 baseline BM25 的结果。

在离线评测中，如果 query 呈现长自然语言描述或语义改写特征，semantic gate 会额外触发 BGE 向量召回，再和 optimized 结果做低权重 RRF 融合。这个设计的重点是“条件触发”，不是让 embedding 无条件干预所有查询。

## 第 12 页：查询意图识别与动态权重

第一个核心优化是查询意图识别和动态字段权重。

系统会把 query 分成姓名查询、研究方向查询、学院加方向查询、论文成果查询、职称加方向查询和口语化查询等类型。

比如“计科院做知识图谱的老师”，系统会识别学院是计算机科学与技术学院，同时把“知识图谱”作为研究方向。此时学院不再只是普通关键词，而是 hard filter；研究方向字段权重也会提高。

这种做法能让不同查询类型使用更匹配的排序策略。

从消融结果看，adaptive 阶段带来了主要提升，这说明查询意图识别、动态字段权重和学院 hard filter 是我们主模型里最关键的部分。

## 第 13 页：查询扩展与模糊姓名匹配

第二个优化是查询扩展和模糊姓名匹配。

查询扩展用于解决用户用词和网页用词不完全一致的问题。比如 NLP 可以扩展到自然语言处理、机器翻译、信息抽取；知识图谱可以扩展到图谱、实体关系、语义表示等。

模糊姓名匹配则用于处理姓名输入不完整或有轻微错误的情况，提高姓名类查询的鲁棒性。

不过我们也注意到，查询扩展不是全局一定提升，所以它需要结合查询意图使用，而不是无条件扩大 query。

## 第 14 页：论文成果类 RRF 融合

第三个优化是论文成果类 RRF 融合。

教师主页中的论文成果字段通常很长，里面包含论文标题、期刊会议、引用指标和项目成果等内容。长文本信息丰富，但也容易引入噪声。

所以当 query 包含“论文”“成果”“发表”“ACL”“SCI”等词时，系统会融合两路排序结果：optimized fielded BM25 和 baseline BM25。

融合方法是 RRF，也就是根据候选在不同排序列表中的名次进行融合。这样可以把多路结果中都比较稳定的候选排到前面。

需要说明的是，当前 RRF 只用于论文成果类查询，不是所有 query 都使用。

这个限制本身也很重要。我们没有把 RRF 无条件加到所有查询上，因为多路融合会带来额外噪声。只在论文成果类 query 上触发，能更符合它的使用场景。

## 第 15 页：BGE 条件触发语义补充召回

第四个优化是 BGE 条件触发语义补充召回，这也是我们后面补充得比较重要的一部分。

一开始我们希望用 BGE embedding 解决字面不匹配的问题。比如用户不一定直接输入网页里的关键词，而是用“做文本理解和信息抽取的老师”这类自然语言描述。这类 query 只靠 BM25 可能会漏掉一些语义相关结果。

但是实验后我们发现，纯 BGE semantic 并不适合作为主排序器。它在主测试集上低于 optimized，原因是教师主页数据规模不大、字段结构很强，很多查询其实依赖姓名、学院、论文关键词等精确匹配。通用 embedding 容易把“主题相近但不是标注目标”的老师召回到 Top-5。

所以我们的改进不是简单地“加 embedding”，而是做成 conditional_semantic。系统默认仍然使用 optimized；只有 semantic_gate 判断 query 属于长自然语言或语义改写类查询时，才触发 BGE Top-50 补充召回，然后和 optimized Top-50 做低权重 RRF 融合。

这个模块对应的代码主要包括 `semantic_index.py`、`semantic_gate.py` 和 `evaluate_search.py` 里的 `conditional_semantic` 模式。它的研究意义在于：我们保留 BM25 在结构化精确查询上的稳定性，同时让 BGE 只在它更可能有效的语义泛化场景中发挥作用。

## 第 16 页：评测集与评价指标

为了证明优化是否有效，我们构建了正式评测集。

评测集包含 48 条 query 和 216 条 qrels 标注。query 类型包括姓名查询、模糊姓名、研究方向、学院加方向、论文成果和口语化复合查询等。另有 10 条语义泛化 query 和 65 条语义 qrels，专门用于验证语义改写场景。

评价指标主要有三个：Precision@5 衡量 Top-5 中相关教师比例；MRR 衡量第一个相关结果是否靠前；NDCG@5 同时考虑相关性等级和排序位置。

此外，我们也记录平均查询耗时，判断系统是否适合交互式使用。

这里需要强调，评测集覆盖了姓名、模糊姓名、研究方向、学院加方向、论文成果和口语化复合查询。这样能避免系统只在少数简单 query 上表现好。

## 第 17 页：整体实验结果

这一页是 baseline 和 optimized 的整体对比。

根据项目文档记录的正式 48 query 消融评测结果，baseline 的 P@5 是 0.4583，MRR 是 0.7899，NDCG@5 是 0.6300；optimized 的 P@5 是 0.5708，MRR 是 0.9219，NDCG@5 是 0.8186。

可以看到，optimized 在三个核心指标上都有提升。P@5 提升说明 Top-5 中相关教师比例更高；MRR 提升说明第一个相关结果更容易排在前面；NDCG@5 提升说明整体排序质量更好。

另外，后续的 conditional_semantic 门控语义模式在正式主测试集上与 optimized 指标持平，说明它没有破坏主系统的稳定性。

耗时方面，本次复跑中 baseline 约 4.19 ms，optimized 约 11.21 ms，仍然是毫秒级，可以支持前端交互式检索。BGE 相关模式的耗时会受到模型加载和缓存影响，所以汇报中更建议把重点放在 P@5、MRR 和 NDCG@5。

因此这一页的结论是：optimized 在相关性指标上明显优于 baseline，同时没有牺牲交互速度。

## 第 18 页：消融实验与语义向量探索

消融实验能帮助我们判断每个模块的作用。

实验显示，字段级 BM25 单独使用时不一定提升，说明字段建模虽然可解释，但还需要和查询意图结合。加入 fuzzy 后，姓名类查询效果改善明显。查询扩展对部分口语化和领域词 query 有帮助，但不是全局单独提升。

真正带来主要提升的是 adaptive，也就是 fuzzy、扩展、查询意图识别、动态字段权重和学院 hard filter 的组合。optimized 再加入 paper-only RRF，对论文成果类查询进一步补强。

这里我建议汇报时不要回避 fielded 和 expansion 单独不一定提升。这个结果反而说明我们的实验是可信的：不是每个模块都被包装成“必然提升”，而是通过消融判断它应该在哪些场景、以什么组合方式使用。

另外，我们也尝试过 embedding 语义向量召回。原本希望通过 BGE embedding 解决语义相近但字面不匹配的问题，但项目文档中的结果显示，纯 semantic-BGE 低于 optimized，低权重 hybrid 接近 optimized 但仍略低，语义权重过高还会引入噪声。

所以我们没有把 embedding 作为无条件的默认主排序器，而是进一步做了门控，也就是 conditional_semantic。它默认使用 optimized，只在长自然语言或语义改写类 query 上触发 BGE 补充召回，并与 optimized 做加权 RRF。

从项目记录看，conditional_semantic 在正式主测试集上与 optimized 持平；在 10 条语义泛化补充测试集上，P@5 从 optimized 的 0.4600 提升到 0.5000，NDCG@5 从 0.5437 提升到 0.5699。这个结果说明 embedding 不是完全没用，关键是要有门控，避免它影响姓名、论文成果和短关键词这类 BM25 更稳定的查询。

所以 BGE 这一部分最适合的汇报口径是：它不是主模型的替代品，而是一个受控的语义泛化补充模块。我们通过实验确定它的使用边界，这比单纯说“用了深度学习模型”更扎实。

## 第 19 页：系统实现与前端展示

系统实现方面，后端使用 FastAPI，并采用注册式检索引擎设计。

目前后端支持 `bm25` 和 `optimized` 两种主要引擎，前端可以通过参数切换。后端提供搜索接口、教师详情接口和统计接口。

前端使用 Vue、Pinia 和 Element Plus，实现了搜索框、字段选择、Top-K 设置、BM25 和优化检索切换、结果卡片以及教师详情展示。

这部分让系统不仅能在命令行中评测，也能作为一个可交互系统进行演示。

## 第 20 页：完整使用示例

这一页可以做现场演示。

比如我们输入“计科院做知识图谱的老师”，选择“优化检索”，系统会先把“计科院”归一到计算机科学与技术学院，再把“知识图谱”作为研究方向进行检索，并返回相关教师。

如果时间允许，可以切换到基础 BM25 对比同一个 query，观察 optimized 在学院识别和方向排序上的优势。

这个例子能比较直观地展示：我们的系统不是简单关键词匹配，而是会结合查询意图进行检索。

## 第 21 页：总结与展望

最后总结一下。

## 可能被问到的问题

### 为什么不用 embedding 作为主模型？

我们确实尝试过 BGE embedding，但项目文档记录显示，纯语义召回在主测试集上低于 optimized。原因可能是教师主页数据规模较小、字段结构较强，很多查询依赖姓名、学院和论文字段的精确匹配。通用 embedding 容易引入主题相近但并非真正相关的教师。

所以最终做法不是完全放弃 embedding，而是做成 conditional_semantic：默认走 optimized，只有 query 符合语义改写或自然语言描述特征时才触发 BGE 补充召回。这个门控版本在主测试集上不破坏 optimized 表现，在语义泛化补充测试集上有小幅提升。

### optimized 的核心贡献是什么？

核心不是单个模块，而是自适应混合检索框架。系统先识别 query 意图，再调整字段权重，并结合查询扩展、模糊姓名、学院 hard filter 和论文成果类 RRF。实验中 adaptive 和 optimized 的提升说明，这种组合策略是有效的。

### 为什么字段级 BM25 单独不一定提升？

因为教师主页字段存在缺失和不均衡。如果没有查询意图控制，固定字段权重可能会放大字段缺失的问题。字段级 BM25 是可解释建模基础，但需要和动态权重、fuzzy、扩展等模块结合才更稳定。

## 交接给 PPT 制作者的提醒

### 最应该放大的三件事

1. **真实数据工作量**：5 个学院、283 条教师文档、动态入口、模板差异、清洗和字段结构化。
2. **模型不是堆叠**：optimized 是按 query 意图自适应；BGE 是经过失败实验后做成条件触发。
3. **结果有消融支撑**：48 条主测试 query、216 条主 qrels、10 条语义泛化 query、65 条语义 qrels，指标覆盖 P@5、MRR、NDCG@5。

### 不建议写成的说法

1. 不要写“前端有独立 BGE 按钮”或“所有 optimized 查询都会使用 BGE”。当前前端演示主要是 `bm25` 和 `optimized`，BGE 是 optimized 后端内部的条件触发补充路径。
2. 不要写“纯 BGE 全面超过 BM25”。实际结果是 conditional_semantic 在语义泛化子集提升，在主测试集保持不下降。
3. 不要写“所有优化模块单独都提升”。消融显示部分模块需要和查询意图、动态权重组合使用。
4. 不要把数据采集写成完全自动化。当前是 seed 驱动，优点是稳定可复现，后续可扩展。
