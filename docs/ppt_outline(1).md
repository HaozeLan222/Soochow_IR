# 苏州大学导师信息检索系统汇报 PPT 大纲

本文档用于规划小组汇报 PPT，目标页数约 21 页。整体思路是减少数据部分篇幅，把重点放在检索模型、优化策略、评测结果和系统展示上，让汇报更像一个完整的信息检索课程项目，而不是单纯的数据采集项目。

> 数值口径：本文中的数据规模和指标已于 2026-06-13 从本地 JSONL 与 `scripts/evaluate_search.py` 重新统计/复跑验证。相关性指标可直接汇报；耗时会受机器、缓存、BGE 模型加载状态影响，建议只作为量级参考。

## 汇报主线

本项目围绕一个核心问题展开：

> 面向苏州大学导师主页这类半结构化、噪声较多、查询意图复杂的数据，如何构建一个可用、可评测、可优化的垂直信息检索系统？

建议汇报时反复扣住以下几点：

- 数据来自真实教师主页，不是手工构造样例。
- 系统不仅能检索，还能比较 baseline 和 optimized 的效果。
- 优化不是简单堆功能，而是针对不同查询意图做自适应处理。
- BGE 语义向量不是简单堆模型，而是经过失败分析后设计成“条件触发的补充召回”，体现模型选择和误差控制。
- 有正式 query/qrels 和评价指标，能证明优化效果。
- 最终有前后端系统，可现场演示。

## 建议重点突出的创新点

建议把创新点组织为“数据工程 + 自适应检索 + 鲁棒性优化 + 条件式语义增强 + 可量化评测”五层，而不是只罗列功能：

| 层次 | 创新点 | 汇报口径 |
|---|---|---|
| 数据层 | 面向真实教师主页的结构化解析 | 从动态入口、模板差异和噪声网页中构建 283 条教师 JSONL 文档，为字段检索提供基础。 |
| 检索层 | 查询意图识别与动态字段权重 | 根据姓名、学院、研究方向、论文成果、职称等 query 类型调整字段权重，而不是所有查询共用固定权重。 |
| 鲁棒性层 | 查询扩展、模糊姓名、学院 hard filter、paper-only RRF | 分别解决同义表达、姓名输入误差、学院约束和论文成果长文本噪声问题。 |
| 语义增强层 | BGE 条件触发补充召回 | 先验证纯语义召回和无条件 hybrid 的不足，再通过 semantic gate 只在语义改写 query 上触发 BGE，保持主测试集稳定并提升语义泛化集。 |
| 评测层 | query/qrels + 消融实验 | 用 48 条主测试 query、216 条主 qrels，以及 10 条语义泛化 query、65 条语义 qrels 证明各模块效果，而不是只展示主观样例。 |

## 建议采用的汇报叙事线

不要把 PPT 讲成“我们写了哪些脚本”的流水账。更推荐讲成一个完整的信息检索实验闭环：

```text
真实网页数据难处理
  -> 结构化教师文档
  -> 建立 BM25 baseline
  -> 观察 baseline 在复合 query 上不足
  -> 针对不同错误类型设计 optimized
  -> 引入 BGE，但通过实验发现不能无条件替代 BM25
  -> 设计 semantic_gate 条件触发补充召回
  -> 用主测试集和语义泛化测试集做量化验证
  -> 落到前后端系统展示
```

可以反复强调三条证据链：

1. **工作量证据**：5 个学院、283 条教师文档、真实 HTML、动态入口、模板差异、JSONL 清洗。
2. **算法思考证据**：不是简单 BM25，也不是盲目加深度模型，而是按查询意图做动态策略，BGE 也经过失败分析后做条件触发。
3. **实验可信证据**：48 条主 query、216 条主 qrels、10 条语义泛化 query、65 条语义 qrels、P@5/MRR/NDCG@5 和消融表。

## 第 2 页：项目目标与核心问题

### 标题

项目目标与核心问题

### 内容

- 构建面向苏州大学导师信息的垂直检索系统。
- 基于苏州大学 5 个学院的真实教师主页数据，支持导师信息查询。
- 支持按教师姓名、学院、研究方向、论文成果等方式查询。
- 从真实网页数据出发，完成数据处理、索引构建、检索排序、结果展示和效果评测。

### 讲述重点

普通搜索引擎不一定适合导师查找场景。用户更关心的是：

- 谁研究某个方向？
- 某个学院有哪些相关导师？
- 某位老师的主页、研究方向和论文成果是什么？
- 输入口语化查询时，系统能否返回真正相关的导师？

具体覆盖的 5 个学院和数据规模会在后面的“数据来源与规模”部分展开。

这里建议把项目目标说成“垂直领域 IR 系统”，而不是“搜索网页”。我们的对象是导师主页，难点是字段半结构化、页面噪声多、用户 query 类型复杂，所以后面才需要字段建模、查询意图识别和语义补充召回。

## 第 3 页：系统整体架构

### 标题

系统整体架构

### 内容

建议画一张 pipeline 图：

```text
教师主页 HTML
  -> 网页解析
  -> 数据清洗
  -> 教师 JSONL 文档
  -> 索引构建
  -> 检索模型层
       -> BM25 baseline（对照模型）
       -> 自适应混合检索 optimized（本文主模型）
       -> conditional_semantic（离线评测：BGE 条件触发补充召回）
  -> FastAPI 后端
  -> Vue 前端
  -> 评测与展示
```

### 讲述重点

这一页只讲全局视角，不展开细节。需要强调：BM25 是基础对照模型，真正作为本项目优化主线的是自适应混合检索模型，它在字段级 BM25 的基础上加入查询意图识别、动态字段权重、查询扩展、模糊姓名匹配、学院 hard filter 和论文成果类 RRF 融合。除此之外，项目还实现了 conditional_semantic 离线评测模式：在 optimized 稳定主排序基础上，只对语义改写类 query 触发 BGE 补充召回，避免无条件 embedding 带来的噪声。后续页面会分别解释这些模块。

如果只画一张架构图，建议把“评测”也画进闭环：搜索结果不是只靠主观观察，而是回到 query/qrels 指标中验证。

## 第 5 页：数据采集与解析策略

### 标题

数据采集与解析策略

### 内容

- 使用 seed 文件维护教师主页 URL。

可以补一句：这里体现的是工程取舍。完全自动发现所有教师主页当然更理想，但课程项目更需要稳定、可复现、能扩展到其他学院的数据入口。

## 第 8 页：Baseline 检索模型

### 标题

Baseline：BM25 检索模型

### 内容

- 使用 BM25 作为基础排序模型。
- 中文分词优先使用 jieba。
- 无 jieba 时使用字符 n-gram 兜底。
- 支持姓名精确匹配。
- 支持学院、研究方向、论文、职称等字段过滤。
- 使用基础字段加权：
  - 姓名权重最高。
  - 研究方向权重较高。
  - 论文、简介、学院、全文参与综合排序。

### 讲述重点

Baseline 是后续优化的对照组。它能完成基本检索，但对复杂查询意图适应不足。

这里可以明确说：baseline 的作用不是“差模型”，而是一个可复现、可量化的参照。后续所有优化都要和它比较。

## 第 10 页：为什么需要优化检索

### 标题

为什么需要优化检索

### 内容

用户查询并不都是同一类：

| 查询例子 | 查询意图 |
|---|---|
| 周国栋 | 姓名查询 |
| 自然语言处理 | 研究方向查询 |
| 计科院 做知识图谱的老师 | 学院 + 方向 + 口语化查询 |
| 发表 ACL 论文的导师 | 论文成果查询 |
| 数学学院 动力系统 | 学院 + 研究方向查询 |

### 讲述重点

单一 BM25 权重很难同时适配所有查询。比如姓名查询应该重视 `name` 字段，研究方向查询应该重视 `research` 字段，论文查询应该重视 `papers` 字段。

这一页是模型设计的转折点：从“统一关键词匹配”转向“先理解 query，再选择检索策略”。

## 第 11 页：Optimized 检索总体流程

### 标题

Optimized：自适应混合检索

### 内容

建议画成流程图：

```text
用户 query
  -> 查询意图识别
  -> 学院别名归一
  -> 去除口语 filler
  -> 动态字段权重
  -> 查询扩展
  -> 模糊姓名匹配
  -> 学院 hard filter
  -> paper 类 RRF 融合
  -> semantic_gate 判断是否触发 BGE 补充召回（离线评测）
  -> Top-K 结果
```

### 讲述重点

强调 optimized 不是单一算法替换，而是一套面向导师检索场景的自适应策略。BGE 部分也不是直接替换 BM25，而是通过 gate 判断 query 是否需要语义泛化；这体现了“先分析错误，再控制触发条件”的模型迭代思路。

## 第 12 页：优化点一：查询意图识别与动态权重

### 标题

查询意图识别与动态字段权重

### 内容

系统会识别：

- 姓名查询。
- 研究方向查询。
- 学院 + 方向查询。
- 论文成果查询。
- 职称 + 方向查询。
- 口语化查询。

示例：

| Query | 处理方式 |
|---|---|
| 计科院 做知识图谱的老师 | 识别学院为计算机科学与技术学院，提高 research 权重 |
| 数学学院 动力系统 | 学院 hard filter + 研究方向检索 |
| 发表 ACL 论文的导师 | 提高 papers 字段权重 |

### 讲述重点

学院不再只是普通关键词，而是转为 hard filter。这样可以减少其他学院教师误入结果。

## 第 13 页：优化点二：查询扩展与模糊姓名匹配

### 标题

查询扩展与模糊姓名匹配

### 内容

查询扩展例子：

| 原始词 | 扩展词 |
|---|---|
| NLP | 自然语言处理、机器翻译、信息抽取 |
| 知识图谱 | 图谱、实体关系、语义表示 |
| 机器学习 | 深度学习、模型、算法 |

模糊姓名匹配：

- 支持近似输入。
- 提升姓名查询的鲁棒性。
- 对 fuzzy_name 类 query 有明显帮助。

### 讲述重点

查询扩展适合解决“用户词和网页词不完全一致”的问题；模糊姓名匹配适合解决姓名输入不完整或有轻微误差的问题。

## 第 14 页：优化点三：论文成果类 RRF 融合

### 标题

论文成果类查询的 RRF 融合

### 内容

针对包含以下词的 query：

- 论文
- 成果
- 发表
- 期刊
- 会议
- ACL
- SCI

系统会对两路结果做 RRF 融合：

```text
optimized fielded BM25 结果
        +
baseline BM25 结果
        ->
RRF 融合排序
```

### 讲述重点

论文成果字段通常文本长、噪声多，单一路径容易漏召回。RRF 可以把不同排序来源中的稳定候选融合起来，提升论文成果类查询效果。

## 第 15 页：优化点四：BGE 条件触发语义补充召回

### 标题

BGE 条件触发语义补充召回

### 内容

为什么不是直接使用 embedding 替代 BM25：

| 方案 | 主测试集表现 | 问题 |
|---|---|---|
| 纯 BGE semantic | 低于 optimized | 容易召回主题相近但标注不相关的教师 |
| 无条件 hybrid semantic | 接近但未超过 optimized | 对姓名、学院、论文短词等精确查询会引入噪声 |
| conditional_semantic | 主测试集不下降，语义泛化集提升 | 只在需要语义泛化时触发 |

实现思路：

```text
用户 query
  -> semantic_gate 判断是否属于长自然语言/语义改写类查询
  -> 默认使用 optimized
  -> 触发时召回 BGE Top-50
  -> optimized Top-50 + BGE Top-50 加权 RRF 融合
```

关键代码：

- `suda_ir/ir/semantic_index.py`：BGE 向量索引与缓存。
- `suda_ir/ir/semantic_gate.py`：触发规则，避开姓名、论文、短关键词等 BM25 更稳定的场景。
- `scripts/evaluate_search.py`：`conditional_semantic` 离线评测模式。

### 讲述重点

这一页要把 BGE 讲成一次完整的研究迭代：我们先尝试纯语义召回，发现它并不适合作为主排序器；再尝试无条件 hybrid，发现语义权重过高会引入噪声；最后设计 `semantic_gate`，只在自然语言改写类 query 上做补充召回。它的价值不是“全面超过 BM25”，而是在保持主测试集稳定的同时，提升语义泛化 query 的表现。

## 第 17 页：整体实验结果

### 标题

Baseline 与 Optimized 整体对比

### 内容

核心结果表：

| 模式 | P@5 | MRR | NDCG@5 | 平均耗时 |
|---|---:|---:|---:|---:|
| baseline | 0.4583 | 0.7899 | 0.6300 | 约 4.19 ms |
| optimized | 0.5708 | 0.9219 | 0.8186 | 约 11.21 ms |
| conditional_semantic | 0.5708 | 0.9219 | 0.8186 | 离线评测，受 BGE 加载/缓存影响 |

### 讲述重点

可以这样表述：

- P@5 从 0.4583 提升到 0.5708，说明 Top-5 结果中相关教师比例提高。
- MRR 从 0.7899 提升到 0.9219，说明第一个相关结果更容易排在前面。
- NDCG@5 从 0.6300 提升到 0.8186，说明整体排序质量明显提升。
- baseline 和 optimized 平均耗时仍在毫秒级，本次复跑分别约 4.19 ms 和 11.21 ms，适合交互式检索；BGE 相关模式受模型加载和缓存影响，不建议用单次耗时作为主要结论。
- `conditional_semantic` 在正式主测试集上与 optimized 持平，说明门控策略没有破坏主系统稳定性。

这里建议不要把 `conditional_semantic` 说成“主测试集继续提升”，因为实际指标是持平。更 solid 的说法是：它在主测试集上不破坏 optimized，在语义泛化补充集上有提升，因此适合作为受控补充模块。

## 第 18 页：消融实验分析

### 标题

消融实验与语义向量探索

### 内容

| 版本 | P@5 | MRR | NDCG@5 |
|---|---:|---:|---:|
| baseline | 0.4583 | 0.7899 | 0.6300 |
| fielded | 0.4417 | 0.7378 | 0.5992 |
| fielded + fuzzy | 0.4625 | 0.8420 | 0.7034 |
| fielded + expand | 0.4458 | 0.7795 | 0.6128 |
| adaptive | 0.5500 | 0.9219 | 0.8052 |
| optimized | 0.5708 | 0.9219 | 0.8186 |
| conditional_semantic | 0.5708 | 0.9219 | 0.8186 |

语义向量探索结果，来自项目交接文档记录：

| 版本 | P@5 | MRR | NDCG@5 | 结论 |
|---|---:|---:|---:|---|
| semantic-BGE | 0.3250 | 0.7438 | 0.5691 | 纯语义召回低于 BM25 hybrid |
| hybrid_semantic-BGE, weight=0.05 | 0.5542 | 0.9219 | 0.8113 | 接近 optimized，但仍略低 |
| hybrid_semantic-BGE, weight=0.6 | 0.4583 | 0.8958 | 0.7142 | 语义权重过高会引入噪声 |

语义泛化补充测试集：

| 模式 | P@5 | MRR | NDCG@5 |
|---|---:|---:|---:|
| optimized | 0.4600 | 0.8833 | 0.5437 |
| conditional_semantic | 0.5000 | 0.9000 | 0.5699 |

### 讲述重点

- 字段级 BM25 是可解释建模基础，但单独使用不一定提升。
- fuzzy 对姓名类查询有效。
- 查询扩展对部分口语化和领域词查询有效，但不是全局单独提升。
- adaptive 是核心提升来源。
- paper-only RRF 对论文成果类 query 有额外提升。
- 项目中曾尝试使用 BGE embedding 做语义向量召回，希望解决语义相近但字面不匹配的问题。
- 但在正式 48 query 主测试集上，纯语义召回效果低于 optimized，无条件混合语义召回也没有超过当前主模型。
- 原因可能是教师主页数据规模较小、字段结构较强，而通用 embedding 容易把“主题相近但不是标注相关目标”的教师引入 Top-5。
- 因此后续加入了 `semantic_gate`，只在长自然语言/语义改写类 query 上触发 BGE 补充召回，这就是 `conditional_semantic`。
- `conditional_semantic` 在正式主测试集上不破坏 optimized 表现，在 10 条语义泛化补充测试集上相对 optimized 有小幅提升。
- 需要注意：当前前端/后端演示只提供 `bm25` 和 `optimized` 两个入口；BGE 不单独作为前端按钮，而是作为 `optimized` 后端内部的可选门控补充召回路径。

这一页建议重点讲“不是每个模块单独都提升”。这反而能体现实验扎实：fielded 和 expand 单独不一定提升，说明我们不是为了凑创新点，而是通过消融判断每个模块应放在什么位置。最终贡献是自适应组合策略，而不是简单功能堆叠。

### 可选补充说明

### 最建议强调的最终结论


### 不建议在汇报中夸大的说法
