# 数据清洗工作说明

本文档说明苏州大学导师信息检索系统中数据清洗阶段的主要操作、处理流程和关键技术细节。清洗目标不是简单地把网页正文保存下来，而是将格式不统一、噪声较多的教师主页 HTML 转换为字段清晰、结构稳定、适合检索建模的 JSONL 数据。

## 一、清洗目标

导师主页来自不同学院、不同模板和不同教师个人页面，页面结构存在较大差异。原始 HTML 中通常混杂了导航栏、页脚版权、访问量、栏目菜单、无效占位内容、重复文本、英文主页内容、实验室主页内容以及格式断裂的论文和研究方向文本。

因此，本项目的数据清洗主要完成以下目标：

- 将本地 HTML 文件批量解析为统一的教师文档格式。
- 抽取教师姓名、学院、职称、研究方向、论文成果、个人简介等核心字段。
- 删除导航栏、页脚、访问量、版权信息等网页噪声。
- 修复网页转文本后产生的断行、断词、编号错乱、标点分离等问题。
- 保留教师主页中公开展示的邮箱、电话等联系信息，使其能够参与导师信息检索。
- 对字段缺失、页面质量较差、疑似非教师主页的数据进行质量标记。
- 生成清洗报告，便于统计数据质量和后续筛选。

## 二、整体流程

数据处理流程分为两个阶段。

第一阶段是 HTML 到 JSONL 的初步解析。脚本 `scripts/parse_handoff_html_to_jsonl.py` 会遍历本地保存的教师主页 HTML，根据种子文件中的教师姓名、学院、URL 信息进行校正，然后调用解析器抽取正文和字段，最后输出 `handoff_teachers.jsonl`。

第二阶段是 JSONL 的二次清洗。脚本 `scripts/clean_teacher_jsonl.py` 会对初步解析得到的教师文档进行进一步规范化处理，包括清理正文、修复字段、推断缺失职称、重组全文字段、评估数据质量，并输出 `handoff_teachers.clean.jsonl` 和对应的清洗报告。

当前数据规模如下：

| 文件 | 数量 | 说明 |
|---|---:|---|
| `handoff_teachers.jsonl` | 296 | 从原始 HTML 初步解析得到的数据 |
| `handoff_teachers.clean.jsonl` | 296 | 二次清洗后的完整数据 |
| `handoff_teachers.filtered.jsonl` | 279 | 去除低质量样本后的数据 |

清洗后质量统计如下：

| 质量等级 | 数量 |
|---|---:|
| high | 199 |
| medium | 80 |
| low | 17 |

## 三、具体清洗操作

### 1. 页面识别与非教师页过滤

原始网页并不一定都是标准教师主页，有些可能是导航页、实验室主页、英文主页或内容极短的异常页面。项目中使用多规则页面判定方法，根据页面中是否出现“教师个人主页”“个人简介”“研究方向”“职称”“电子邮箱”等中文教师页信号，或 `homepage`、`research interests`、`publications`、`professor`、`Soochow University` 等英文教师页信号，判断页面是否可能是教师主页。

如果页面只包含较弱信号，或者存在“学院列表”“教师分类查询”“热点主页”等导航页特征，则会被标记为弱质量或非标准页面。这种做法比直接保存所有 HTML 更稳妥，可以减少后续检索中无关页面对结果排序的干扰。

### 2. 使用种子信息校正元数据

教师姓名和学院有时无法从网页正文中稳定抽取，尤其是部分主页模板中姓名位置不固定，或者网页标题、栏目文本会干扰姓名识别。因此，解析阶段会读取 seed CSV 中的教师姓名、学院和 URL，并用这些可信来源校正文档字段。

如果页面解析出的姓名与 seed 中的姓名不一致，程序会保留解析结果到 `extra["parsed_name"]`，但最终以 seed 中的姓名为准。学院也优先使用本地目录或 seed 信息。这一设计既保证了最终字段准确，又保留了原始解析痕迹，便于后续排查。

### 3. 网页噪声清理

清洗脚本中维护了明确的噪声词表和页脚模式，用来删除常见无效内容。例如：

- “教师个人主页”
- “返回首页”
- “欢迎登录”
- “导航”
- “English”
- “最新上线”
- “相关教师”
- “软件著作”
- “专利”
- “Copyright”
- “All Rights Reserved”
- “版权所有”
- “技术支持”

此外，脚本还会识别访问量样式的数字。例如某些页面会出现单独的数字行，下一行是“访问”，这类内容会被自动删除，避免把访问计数误当作正文。

### 4. 栏目菜单块删除

很多教师主页正文中会混入一整段导航菜单，例如：

```text
个人资料
个人概况
研究领域
开授课程
科研项目
论文
```

这类文本本身不是教师内容，而是页面目录。如果直接进入全文索引，会导致所有教师都带有类似关键词，降低检索区分度。项目中通过 `strip_navigation_blocks` 检测连续出现的栏目标题，当短窗口内出现多个栏目标题时，将其视为导航块整体跳过。

这个处理比较巧妙，因为它不是简单删除所有栏目名。栏目名在真正的正文结构中仍然有用，例如“研究领域”后面跟着具体研究方向时不能删除。程序只删除“连续密集出现的栏目标题块”，从而兼顾去噪和保留结构。

### 5. 字段别名抽取

不同学院页面对同一字段的命名不一致。例如研究方向可能写成：

- “研究方向”
- “研究领域”
- “科研方向”
- “主要研究方向”
- “研究兴趣”
- “目前主要研究方向”

论文成果可能写成：

- “论文”
- “学术论文”
- “发表论文”
- “科研成果”
- “科技成果”
- “代表性成果”

项目中通过字段别名表 `FIELD_ALIASES` 将这些不同写法映射到统一字段。这样即使网页模板不统一，也能尽量抽取到标准化的 `research`、`papers`、`profile` 等字段。

### 6. 文本断裂和排版修复

网页转文本后经常出现大量断裂问题，例如编号、标点、英文单词、化学或数学符号被拆到多行。清洗脚本针对这些问题做了多种修复。

例如，将：

```text
1.
有机太阳能电池
:
新型高分子材料设计
```

修复为：

```text
1. 有机太阳能电池：新型高分子材料设计
```

再例如，将：

```text
有机
-
无机杂化光电器件
```

修复为：

```text
有机-无机杂化光电器件
```

还会处理 `H\n-index`、`10. 0`、`C*-\n代数`、URL 后紧贴中文、Arxiv 编号中出现空格等问题。这些规则看起来细碎，但对检索效果很重要，因为 BM25 和分词器依赖相对干净的 token。如果文本中到处是断开的编号和词语，召回和排序都会受影响。

### 7. 不同字段采用不同清洗策略

项目没有对所有字段使用同一种清洗规则，而是根据字段性质分别处理。

对于 `research` 字段，如果检测到多条短编号研究方向，会将它们合并为更紧凑的分号连接文本，便于作为研究主题参与检索。

对于 `profile` 字段，会重点修复教育经历、工作经历中的日期格式，例如将连续堆在一行的 `2021年8月-至今`、`2016年9月-2021年7月` 自动拆分成多行，提高可读性。

对于 `papers` 字段，会重点修复论文编号、引用指标、Google Scholar、ORCID、H-index、i10-index 等论文相关文本，避免论文列表和统计指标被错误断开。

这种按字段定制规则的方式是清洗中比较高级的地方。它说明清洗不是机械替换，而是结合了导师主页数据的语义特点。

### 8. 缺失字段推断

部分教师主页中字段抽取不完整，例如 `title` 为空，但正文前几行可能出现“教授”“副教授”“讲师”等职称。清洗脚本会在正文、简介、研究方向中尝试推断职称。

研究方向也有兜底推断逻辑。如果标准栏目没有抽到 `research`，程序会从正文中的字段别名或个人简介中的句式中提取。例如个人简介里出现“长期从事……研究”或“主要研究方向……”时，也会作为研究方向候选。

这种多级兜底机制可以提高字段覆盖率，避免因为网页模板不规范就丢失重要检索字段。

### 9. 联系方式保留与格式规范

由于本项目的目标是导师信息检索系统，教师主页中公开展示的邮箱、电话、办公地址等信息本身具有检索价值。例如用户可能希望通过姓名检索导师后查看联系方式，或者通过邮箱、办公室等线索定位教师主页。因此，最终用于检索的清洗数据不将联系方式作为默认删除对象。

在处理邮箱时，系统会对 `name [at] suda.edu.cn`、`name at suda dot edu dot cn` 等混淆写法进行格式标准化，尽量还原为更适合检索的统一文本。也就是说，本项目中的联系方式处理重点是“规范化”和“可检索”，而不是强制脱敏。

### 10. 正文重组

清洗后并不是简单保留原始正文，而是重新构造 `content` 字段。重组顺序大致为：

1. 学院
2. 姓名
3. 职称
4. 个人简介
5. 研究领域
6. 教育经历
7. 工作经历
8. 科研项目
9. 论文
10. 荣誉及奖励
11. 招生信息

这样做的好处是，最终全文字段既包含原网页信息，又具有稳定结构。对于后续 BM25 检索来说，结构化重组后的文本比杂乱网页正文更容易获得稳定排序结果。

## 四、质量评估与过滤

每条清洗后的教师文档都会被打上质量等级和问题标记。质量判断主要参考以下因素：

- 是否有职称。
- 是否有研究方向。
- 是否有个人简介。
- 是否有 URL。
- 是否疑似实验室主页。
- 页面信号是否较弱。
- 正文是否过短。

当前清洗报告中常见问题如下：

| 问题标记 | 数量 | 含义 |
|---|---:|---|
| `missing_profile` | 84 | 缺少个人简介 |
| `very_short_content` | 48 | 正文过短 |
| `missing_title` | 37 | 缺少职称 |
| `missing_research` | 24 | 缺少研究方向 |
| `weak_page_signal` | 13 | 教师页信号较弱 |
| `lab_like_homepage` | 2 | 疑似实验室主页 |
| `missing_url` | 1 | 缺少 URL |

清洗脚本既可以保留全部 296 条数据，也可以通过 `--drop-low-quality` 删除低质量样本，生成 279 条的过滤版数据。这种设计保留了两种实验选择：完整数据可用于覆盖率更高的检索，过滤数据可用于质量更稳定的实验对比。

## 五、清洗中比较高级和巧妙的地方

本项目的数据清洗有几个值得重点说明的设计。

第一，采用“结构化字段 + 全文兜底”的策略。对于能抽取出来的姓名、学院、职称、研究方向、论文、简介，系统会存入独立字段；对于字段抽取失败或网页结构异常的情况，仍然保留清洗后的全文 `content`。这样既支持字段加权检索，也避免信息被过度清洗丢失。

第二，使用 seed 信息进行可信校正。教师姓名、学院和 URL 不完全依赖网页解析，而是结合种子文件进行修正。由于网页正文中可能混入学院名、栏目名、英文名或导航文字，单纯依靠正则猜测容易出错。seed 校正显著提高了关键元数据的准确率。

第三，噪声删除不是简单粗暴的关键词删除。比如栏目标题既可能是导航菜单，也可能是正文结构的一部分。项目中通过“连续栏目标题块”的模式来删除导航，而不是删除所有栏目名，避免误伤真实内容。

第四，针对不同字段设计不同清洗规则。研究方向、个人简介和论文成果的文本结构差异很大。研究方向需要合并短列表，个人简介需要整理时间经历，论文字段需要保留编号、H-index、ORCID、Google Scholar 等特殊格式。字段级清洗让最终数据更符合检索需求。

第五，加入了质量标记机制。清洗过程不仅输出结果，还输出 `clean_quality` 和 `clean_flags`。这使后续实验可以清楚知道哪些样本可能影响检索效果，也可以灵活选择完整数据集或过滤数据集。

第六，清洗规则有单元测试验证。项目中 `tests/test_clean_teacher_jsonl.py` 针对导航/页脚删除、职称推断、断裂编号合并、日期规范化、论文指标修复等场景写了测试。说明清洗规则不是临时手改，而是可以重复运行、可验证的工程化处理。

## 六、总结

总体来看，本项目的数据清洗工作不仅完成了基础的格式转换和去噪，还针对导师主页这一特定数据源做了较多定制化处理。清洗过程中兼顾了字段抽取、联系方式保留、文本规范化、质量评估和检索友好性。最终得到的 JSONL 数据结构清晰、字段完整度较高，并且保留了清洗质量报告，能够支持后续 BM25 检索、字段加权检索、模糊查询和语义检索等实验。
