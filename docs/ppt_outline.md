# 苏州大学导师信息检索系统汇报 PPT 大纲

本文档用于规划小组汇报 PPT，目标页数约 20 页。整体思路是减少数据部分篇幅，把重点放在检索模型、优化策略、评测结果和系统展示上，让汇报更像一个完整的信息检索课程项目，而不是单纯的数据采集项目。

## 汇报主线

本项目围绕一个核心问题展开：

> 面向苏州大学导师主页这类半结构化、噪声较多、查询意图复杂的数据，如何构建一个可用、可评测、可优化的垂直信息检索系统？

建议汇报时反复扣住以下几点：

- 数据来自真实教师主页，不是手工构造样例。
- 系统不仅能检索，还能比较 baseline 和 optimized 的效果。
- 优化不是简单堆功能，而是针对不同查询意图做自适应处理。
- 有正式 query/qrels 和评价指标，能证明优化效果。
- 最终有前后端系统，可现场演示。

## 第 2 页：项目目标与核心问题

### 标题

项目目标与核心问题

### 内容

- 构建面向苏州大学导师信息的垂直检索系统。
- 基于苏州大学 5 个学院的真实教师主页数据，支持导师信息查询。
- 支持按教师姓名、学院、研究方向、论文成果等方式查询。
- 从真实网页数据出发，完成数据处理、索引构建、检索排序、结果展示和效果评测。

### 讲述重点

普通搜索引擎不一定适合导师查找场景。用户更关心的是：

- 谁研究某个方向？
- 某个学院有哪些相关导师？
- 某位老师的主页、研究方向和论文成果是什么？
- 输入口语化查询时，系统能否返回真正相关的导师？

具体覆盖的 5 个学院和数据规模会在后面的“数据来源与规模”部分展开。

## 第 3 页：系统整体架构

系统整体架构

### 内容

建议画一张 pipeline 图：

```text
教师主页 HTML
  -> 网页解析
  -> 数据清洗
  -> 教师 JSONL 文档
```

## 第 5 页：数据采集与解析策略

### 标题

数据采集与解析策略

### 内容

- 使用 seed 文件维护教师主页 URL。
- 抓取教师详情页 HTML，并保存原始页面。
- 解析姓名、学院、职称、研究方向、论文、简介等字段。
- 使用 seed 中的姓名和学院校正网页解析结果。

### 讲述重点

说明为什么采用 seed 驱动方式：学校入口页不一定能从静态 HTML 中直接解析出完整教师列表，教师主页详情页相对稳定，所以 seed 驱动更适合课程项目的可复现需求。

## 第 6 页：数据处理中的困难与解决方案

### 标题

数据处理困难与解决方案

## 第 8 页：Baseline 检索模型

### 标题

Baseline：BM25 检索模型

### 内容

- 使用 BM25 作为基础排序模型。
- 中文分词优先使用 jieba。
- 无 jieba 时使用字符 n-gram 兜底。
- 支持姓名精确匹配。
- 支持学院、研究方向、论文、职称等字段过滤。
- 使用基础字段加权：
  - 姓名权重最高。
  - 研究方向权重较高。
  - 论文、简介、学院、全文参与综合排序。

### 讲述重点

Baseline 是后续优化的对照组。它能完成基本检索，但对复杂查询意图适应不足。

## 第 10 页：为什么需要优化检索

### 标题

为什么需要优化检索

### 内容

用户查询并不都是同一类：

| 查询例子 | 查询意图 |
|---|---|
| 周国栋 | 姓名查询 |
| 自然语言处理 | 研究方向查询 |
| 计科院 做知识图谱的老师 | 学院 + 方向 + 口语化查询 |
| 发表 ACL 论文的导师 | 论文成果查询 |
| 数学学院 动力系统 | 学院 + 研究方向查询 |

### 讲述重点

单一 BM25 权重很难同时适配所有查询。比如姓名查询应该重视 `name` 字段，研究方向查询应该重视 `research` 字段，论文查询应该重视 `papers` 字段。

## 第 11 页：Optimized 检索总体流程

### 标题

Optimized：自适应混合检索

### 内容

建议画成流程图：

```text
用户 query
  -> 查询意图识别
  -> 学院别名归一
  -> 去除口语 filler
  -> 动态字段权重
  -> 查询扩展
  -> 模糊姓名匹配
  -> 学院 hard filter
  -> paper 类 RRF 融合
  -> Top-K 结果
```

### 讲述重点

强调 optimized 不是单一算法替换，而是一套面向导师检索场景的自适应策略。

## 第 12 页：优化点一：查询意图识别与动态权重

### 标题

查询意图识别与动态字段权重

### 内容

系统会识别：

- 姓名查询。
- 研究方向查询。
- 学院 + 方向查询。
- 论文成果查询。
- 职称 + 方向查询。
- 口语化查询。

示例：

| Query | 处理方式 |
|---|---|
| 计科院 做知识图谱的老师 | 识别学院为计算机科学与技术学院，提高 research 权重 |
| 数学学院 动力系统 | 学院 hard filter + 研究方向检索 |
| 发表 ACL 论文的导师 | 提高 papers 字段权重 |

### 讲述重点

学院不再只是普通关键词，而是转为 hard filter。这样可以减少其他学院教师误入结果。

## 第 13 页：优化点二：查询扩展与模糊姓名匹配

### 标题

查询扩展与模糊姓名匹配

### 内容

查询扩展例子：

| 原始词 | 扩展词 |
|---|---|
| NLP | 自然语言处理、机器翻译、信息抽取 |
| 知识图谱 | 图谱、实体关系、语义表示 |
| 机器学习 | 深度学习、模型、算法 |

模糊姓名匹配：

- 支持近似输入。
- 提升姓名查询的鲁棒性。
- 对 fuzzy_name 类 query 有明显帮助。

### 讲述重点

查询扩展适合解决“用户词和网页词不完全一致”的问题；模糊姓名匹配适合解决姓名输入不完整或有轻微误差的问题。

## 第 14 页：优化点三：论文成果类 RRF 融合

### 标题

论文成果类查询的 RRF 融合

### 内容

针对包含以下词的 query：

- 论文
- 成果
- 发表
- 期刊
- 会议
- ACL
- SCI

系统会对两路结果做 RRF 融合：

```text
optimized fielded BM25 结果
        +
baseline BM25 结果
        ->
RRF 融合排序
```

### 讲述重点

论文成果字段通常文本长、噪声多，单一路径容易漏召回。RRF 可以把不同排序来源中的稳定候选融合起来，提升论文成果类查询效果。

## 第 16 页：整体实验结果

### 标题

Baseline 与 Optimized 整体对比

### 内容

核心结果表：

| 模式 | P@5 | MRR | NDCG@5 | 平均耗时 |
|---|---:|---:|---:|---:|
| baseline | 0.4583 | 0.7899 | 0.6300 | 4.74 ms |
| optimized | 0.5708 | 0.9219 | 0.8186 | 7.57 ms |

### 讲述重点

可以这样表述：

- P@5 从 0.4583 提升到 0.5708，说明 Top-5 结果中相关教师比例提高。
- MRR 从 0.7899 提升到 0.9219，说明第一个相关结果更容易排在前面。
- NDCG@5 从 0.6300 提升到 0.8186，说明整体排序质量明显提升。
- 平均耗时仍在毫秒级，适合交互式检索。

## 第 17 页：消融实验分析

### 标题

消融实验与语义向量探索

### 内容

| 版本 | P@5 | MRR | NDCG@5 |
|---|---:|---:|---:|
| baseline | 0.4583 | 0.7899 | 0.6300 |
| fielded | 0.4417 | 0.7378 | 0.5992 |
| fielded + fuzzy | 0.4625 | 0.8420 | 0.7034 |
| fielded + expand | 0.4458 | 0.7795 | 0.6128 |
| adaptive | 0.5500 | 0.9219 | 0.8052 |
| optimized | 0.5708 | 0.9219 | 0.8186 |

语义向量探索结果，来自项目交接文档记录：

| 版本 | P@5 | MRR | NDCG@5 | 结论 |
|---|---:|---:|---:|---|
| semantic-BGE | 0.3250 | 0.7438 | 0.5691 | 纯语义召回低于 BM25 hybrid |
| hybrid_semantic-BGE, weight=0.05 | 0.5542 | 0.9219 | 0.8113 | 接近 optimized，但仍略低 |
| hybrid_semantic-BGE, weight=0.6 | 0.4583 | 0.8958 | 0.7142 | 语义权重过高会引入噪声 |

### 讲述重点

- 字段级 BM25 是可解释建模基础，但单独使用不一定提升。
- fuzzy 对姓名类查询有效。
- 查询扩展对部分口语化和领域词查询有效，但不是全局单独提升。
- adaptive 是核心提升来源。
- paper-only RRF 对论文成果类 query 有额外提升。
- 项目中曾尝试使用 BGE embedding 做语义向量召回，希望解决语义相近但字面不匹配的问题。
- 但在正式 48 query 主测试集上，纯语义召回效果低于 optimized，混合语义召回也没有超过当前主模型。
- 原因可能是教师主页数据规模较小、字段结构较强，而通用 embedding 容易把“主题相近但不是标注相关目标”的教师引入 Top-5。
- 因此最终没有把 embedding 作为默认主排序器，而是保留为探索性模块；更稳的主线仍是 optimized 自适应混合检索。

### 可选补充说明

如果汇报时间足够，可以补一句：后续项目中又尝试了 `conditional_semantic`，只在长自然语言或语义改写类 query 上触发 BGE 补充召回，避免它影响姓名、论文成果、短关键词等本来 BM25 更稳定的查询。

## 第 20 页：总结与展望

### 标题

总结与展望

### 已完成工作

- 构建了 5 个学院、283 条教师文档的数据集。
- 实现了 HTML 解析、数据清洗、字段抽取和质量标记。
- 实现了 BM25 baseline 检索系统。
- 实现了自适应 optimized 检索系统。
- 构建了正式 query/qrels 评测集。
- 完成了 P@5、MRR、NDCG@5 和消融实验。
- 实现了 FastAPI 后端和 Vue 前端展示系统。

### 当前不足

- seed 仍有一定人工维护成本。
- 学院和教师覆盖规模还可以继续扩大。
- 语义向量检索已探索，但在主测试集上没有超过 optimized；后续更适合做条件触发的补充召回，而不是默认主排序器。
- 后端存在一份 IR 代码副本，后续可以重构为直接复用主包。

### 后续方向

- 扩展更多学院和教师数据。
- 增加 baseline / optimized 并排对比界面。
- 引入更稳定的语义 rerank 或 LLM rerank。
- 优化教师详情页展示和字段高亮。
- 进一步完善 query 类别和人工标注集。
